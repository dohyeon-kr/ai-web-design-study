# Spacing 작업, 가장 안전한 접근

결론부터: **"부모가 자식들 사이 간격을 책임진다"** 원칙을 잡고 가자. 자식 컴포넌트 자신은 외부 여백을 모른 채로 둔다.

## 왜 이게 안전한가

1. **재사용성이 안 깨진다.** 컴포넌트가 자기 `margin`을 들고 다니면, 다른 맥락에 갖다 놓을 때마다 그 margin이 따라온다. "이 컴포넌트는 항상 위에 16px"이 진실인 경우는 거의 없다 — 맥락마다 다르다. 자식이 margin을 안 가지면 재배치 비용이 0이다.

2. **margin collapsing 함정을 피한다.** 인접 형제의 vertical margin이 합쳐지거나, 자식의 margin이 부모를 뚫고 올라가는 현상은 디버깅 비용이 비싸다. **부모가 `gap` (flex/grid)로 처리하면 collapsing이 아예 없다.**

3. **변경 진폭이 작다.** 간격 조정이 필요할 때 부모 한 곳만 본다. 자식들 곳곳에 흩어진 `margin-top`을 추적할 필요 없음.

## 우선순위 (안전한 순서대로)

1. **`flex` + `gap`** 또는 **`grid` + `gap`** — 가장 안전. 방향성 명확, collapsing 없음, 토큰 한 곳.
2. **부모에 `> * + *` 패턴 (lobotomized owl)** — flex/grid가 어려운 레거시 마크업에서. "사이 간격"의 의미를 정확히 표현.
3. **Stack/Spacer 같은 레이아웃 컴포넌트** — 디자인 시스템이 있다면 이걸 우선. 토큰화·일관성 둘 다 얻는다.
4. (피할 것) **자식의 `margin-top/bottom`** — collapsing·재사용성 둘 다 깨진다. 정말 필요한 경우는 "이 컴포넌트는 이 맥락에서만 산다"가 확실할 때만.

## 추가로 박을 원칙

- **간격 값은 디자인 토큰**(`spacing.sm/md/lg` 등)에서. 매직 넘버 (`16px`) 금지 — 한 곳 바꾸면 전체가 따라와야 한다.
- **방향성을 코드로 드러내라.** `gap`은 양방향이고, `> * + *`는 "사이"만이고, `margin-top`은 "위쪽"이다. 의도와 일치하는 걸 골라야 나중에 읽는 사람이 안 헷갈린다.
- **반응형은 부모에서.** 자식이 viewport별 margin을 들고 다니면 곧 카오스. 부모가 `gap`을 미디어쿼리/컨테이너쿼리로 바꿔준다.

요약하면: **부모가 `gap`으로 간격 책임, 자식은 외부 여백 모름, 값은 토큰으로.** 이 셋만 지키면 spacing 관련 회귀 90%는 안 난다.
